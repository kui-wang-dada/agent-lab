## 合规与曝光边界（2026-05-03 新增，重要）

Kevin 在中国做全栈 + AI 工程师，海外业务为主（Upwork 接单），但有一条不可妥协的合规曝光边界——避免在国内自媒体里暴露"频繁访问境外"的痕迹。

**采用"B 层"边界**：

| 类别 | 做 / 不做 |
|---|---|
| Upwork 接单（海外业务核心） | ✅ 持续做，无变化 |
| 英文版个人网站（同站双语，hreflang 区分） | ✅ 做。作为海外搜索引擎入口，让海外用户自然找到 |
| GitHub 项目持续提交 | ✅ 做。GitHub 在国内技术圈是共识工具 |
| 国内技术视频里展示 GitHub 截图（讨论代码 / README） | ✅ 可以，因为是技术圈共识工具 |
| 海外社交平台（YouTube / dev.to / X） | ❌ 现阶段不开，未来加入时用与国内身份隔离的独立账号 |
| 国内简介挂 GitHub / X / YouTube 等海外平台链接 | ❌ 不挂 |
| 国内视频里展示 Vercel / Twitter / YouTube / dev.to / Reddit 等海外平台访问操作 | ❌ 不展示 |
| 国内视频里演示英文版网站操作 | ❌ 不演示。英文版作为海外搜索引擎入口存在，不主动在国内曝光 |
| 录视频选题里"我用 X 做了 Y"型 case study 中的 X | 必须是国内可访问、且不暗示频繁访问海外的工具 |

**Why**：避免被国内观众或平台审核解读为"频繁访问境外"，规避不必要的法律或平台合规风险。

**注意**：这不是"和海外切割"，是"双轨曝光"——海外业务该做的继续做，国内自媒体作为单独通道走"完全本土化"路线。这也是国内 solo 工程师做海外品牌时常见的工艺。

### 内容公式：case study 而非教程

每条内容套这个模板：
> 我用 [工具/技术] 做了 [真实项目]，过程是 [关键决策/踩坑]，结果是 [可见数据/交付物]。

3 个必须元素：
1. **真实项目**（不是 demo，最好是 Upwork 客户项目脱敏版 / 自己在用的工具）
2. **关键决策瞬间**（你"做不做、怎么做"的判断展现）
3. **可验证结果**（截图、数据、客户反馈、上线链接、GitHub 仓库）

GitHub 项目同样原则：放真实在用的工具，README 写"为什么做 / 自己怎么用 / 效果如何"，不要做教学项目。

**避免的内容形态**：
- ❌ 纯教程（"X 工具入门"、"如何用 Y"）— 同质化严重，吸引不到目标受众
- ❌ 工具评测、行业新闻评论、"N 个思考" — 与个人身份关联弱
- ❌ 翻译/转发别人内容 — 不构建自己档案

### 工作流：文字稿先行

每条长视频先写 800-1500 字文字稿，再录视频。文字稿同时是"未来加入图文平台"时的现成材料库——现阶段不发布，只用作脚本，未来加小红书 / dev.to / 掘金时一稿多用。

国内优先：
1. **小红书** — 长内容源图文版的中文版本，搜索复利平台
2. **掘金** — 国内开发者圈口碑

海外（用独立身份发布，与国内不互相露出）：
3. **YouTube** — 海外视频沉淀，长期 SEO 复利。**重要**：发布身份与国内抖音 / B 站账号不打通，国内自媒体里也不展示"我有 YouTube"

- ❌ **视频号** — 避免矩阵膨胀
- ❌ **LinkedIn** — 1-2 年后视情况加入

## 微产品矩阵流水线（2026-05-03 新增）

### 核心定位

利用 AI 并行槽位（多开 Claude Code / Cursor 窗口）做迷你产品矩阵。**不是为了做出爆款**，是为了：
1. 每个产品 = 一条自媒体内容素材
2. 每个产品 = 一次新技术领域 / 新 AI 工作流的实战
3. 1-2 年累积 = 15-30 个公开可见的产品案例集（专业可信度的硬背书）

**商业化是意外之喜，不是目标**。某个产品跑出来很好，自然变现；跑不出来，下架，下一个，已完成。

### 节奏

- **每 2 周 1 个产品**（不是每周——给"个人经历类内容"留出空间，避免内容同质化和疲劳）
- 全年理论上限 ≈ 25 个产品，实际预期 15-20 个

### 工艺（关键，2026-05-03 共识）

不是"从 0 开始边写边录屏"——那会占满时间。流程是：

1. **第 1-10 天**：AI 并行写产品。Kevin 决定要不要做、UI 风格、关键决策点；Claude Code / Cursor 实际写代码；Kevin review + 部署
2. **第 11-13 天**：写文案（800-1500 字脚本，沿用既定的"文字稿先行"工艺）
3. **第 14 天（周六）**：整体录屏 + 剪辑发布

这套工艺保证：写代码时间用 AI 并行解决，录屏只在固定时间块发生，**不打断主线 Upwork**。

### 四条约束（2026-05-03 共识）

1. **准入门槛**：自己或身边一个具体的人能用一次。筛掉所有 demo 和"想象中的用户"。
   - 通过：给老婆做的小工具、宠物店朋友的预约小程序、自己在用的工具、Upwork 客户副产物
   - 不通过：纯炫技 demo、不知道为谁做的"AI 助手"

2. **技术领域轮换**：每个产品故意换品类，1 年覆盖至少：
   - AI 能力封装（LLM API + UI）
   - Agent / 多步任务 / MCP
   - 数据 / 爬虫 / 分析
   - 集成（飞书 / 微信 / Notion / 抖音 API 等）
   - 嵌入式（浏览器扩展 / VSCode 插件 / CLI）
   - 多模态（图片 / 音频 / 视频）
   - 个人网页 / 内容站 / SEO 类

3. **闭环定义**：每个产品最低线 = **上线（哪怕一键部署）+ 1 篇 README + 1 条视频或图文 = 完成**。完成后可以下架，但必须先达到完成度。**先完成度，再质量**。

4. **内容角度刻意轮换**：避免每条视频都是"我用 Claude Code 1 周做了 X"的同模板。轮换：
   - "我尝试做 X，失败了，原因是..."（反向案例最受欢迎）
   - "X 我用 3 种 AI 方案对比"（横评型）
   - "X 上线 30 天后用户告诉我..."（后置反馈型）
   - "用户找我做 X，我推了，原因是..."（决策型）
   - "做 X 时 Claude Code 让我意外的一步..."（工艺型）

### 统一叙事

虽然产品广泛、品类不重复，但**人是统一的**——观众跟的是"Kevin 这位中国全栈 + AI 工程师并行造各种小产品"这个 IP。可以做宠物预约 + 抖音封面工具 + 个人网页 + 老婆的家务清单 App，看似四个不相关的方向，叙事统一：一个 AI 工程师在并行造东西。

参考形态：levelsio（Pieter Levels）"12 startups in 12 months"。

## 内容节奏与轮换（2026-05-03 新增）

不是所有自媒体内容都是产品 case study。每 2 周 1 条产品视频之外，穿插**个人经历类内容**——这是 IP 真实感的另一面，也降低内容生产疲劳。

### 内容轮换主题池

每 2 周 1 条产品视频 + 1-2 条"非产品"内容，主题池轮换：

- **婚姻 / 家庭**：和老婆怎么协作、家庭决策
- **育儿**：两个儿子的教育观、踩过的坑
- **过去经历**：10 年技术职业生涯的转折点、关键决策、失败回顾
- **远程工作 / 自由职业**：在中国服务海外客户的真实经验
- **AI 工程师的日常**：工具栈选择、Vibe Coding 工作流、效率心得

### 为什么需要这一节

观众跟 IP 不只是跟"作品"，更是跟"人"。levelsio 也写婚姻、生活、世界观——这些内容**不变现，但建立信任**。Kevin 的差异化优势就包含"在中国 / 有家庭 / 全栈背景"这些"人"的维度，扔掉太可惜。

### 双周内容产出节奏（参考）

| 时间段 | 内容 |

| 第 1-10 天 | 产品制作（AI 并行槽位，不挤占 Upwork 主时间块）|

具体节奏可调，关键是 **不勉强自己每周都必须有产品视频**。

## GitHub 开源项目方向（2026-05-03 修订）

GitHub 是微产品矩阵中"代码可分享类"产品的全球分发渠道。不做"炫技 / 造轮子"型项目，所有 GitHub 项目都来自微产品矩阵流水线（即都满足"自己或身边人能用一次"准入门槛）。

技术领域跟着流水线轮换，不预先固定品类。早期参考方向（不限定）：

- **Vibe Coding 工作流模板** — Claude Code starter / MCP server templates / Cursor rules 集
- **Upwork 项目副产物抽象** — 反复造的轮子（RAG pipeline / 多模型中间件等）开源化
- **真实在线运行的小产品** — 自己每天用、放出来给别人用
- **跨平台工具集** — CLI / 浏览器扩展 / VSCode 插件等

国内市场对应的产品分发渠道（与 GitHub 并行）：
- 小程序（微信生态）
- 国内独立站 + 备案
- 抖音小风车 / 留资
- 国内开源（Gitee）—— 有需要时

启动原则：每个产品都要满足"微产品矩阵流水线"的四条约束。

## Upwork 风险评估

- 目前仅一个长期客户，主动选择不扩展
- 判断：半年内客户稳定，风险可控
- 即使突发断单，有 2-3 个月经济缓冲期重新找单
- **新增视角（2026-04）**：1-2 年后 GitHub + YouTube 起来，反而可能反哺 Upwork 时薪上调，或带来海外公司远程 offer

## 约束条件

- 家庭：两个儿子，家庭时间需要保障
- 资金：无外债，资金压力不大
- 偏好：远程办公，时间自由
- 地理：在中国，服务海外客户，有时差
- **新增（2026-04）**：明确不追求短期变现，1-2 年内不主动卖产品
- **新增（2026-05-03）**：合规曝光边界——国内自媒体内容里不展示海外网站 / 访问行为；海外业务（Upwork / 英文版网站 / GitHub）继续做但不与国内身份做明显跨链。详见"合规与曝光边界"节

## 长期方向（1-3 年）（2026-05-03 修订）

以 Upwork 自由职业为基本盘维持稳定收入，同时用 30%+10% 精力做"国内 + 全球"双轨内容积累 **+ 微产品矩阵流水线**（通过 AI 并行槽位实现，不挤占主线 Upwork 时间）。

**1-2 年后的可能形态**（非承诺，是方向）：
- Upwork 时薪从 $25 升到 $50-80（个人品牌背书）
- **微产品矩阵 15-30 个上线产品，技术领域覆盖 6-7 个品类**
- GitHub 1-2 个 1k+ star 项目（来自矩阵中表现最好的）
- YouTube 50-100 条英文视频，订阅 1-5k
- B 站 / 抖音 50-100 条中文视频，国内 AI 工程赛道识别度
- X 关注 3-10k，全球开发者圈有声音
- 海外公司远程 offer / 付费咨询 / 国内合作机会**自然找上门**（双向）

变现方式（届时再选）：接单提价、付费咨询、产品矩阵中跑出来的产品自然变现、付费内容、培训、海外 remote offer。**核心仍是不强求**。

## 复盘 checkpoint（强制）

- **2026-10-27（半年复盘）**：哪个平台在涨？哪个没动？要不要砍？流水线跑顺了吗？
- **2027-04-27（一年复盘）**：J 曲线开始拐了吗？还是要不要切换路线？
- **2028-04-27（两年复盘）**：是否进入下一阶段（变现 / 转型 / 维持）

## 明确不想做的事

- 特别复杂和忙碌的项目
- 占满全部时间、没有弹性的工作安排
- **新增（2026-04）**：
  - TikTok Shop / 抖店实物带货（短期变现导向，与长期事业冲突）
  - 直播带货
  - 接广告陪跑、与开发者人设无关的合作
  - **任何短视频内容生产**（抖音短视频、TikTok、视频号）—— 已明确放弃
  - 为单一平台单独生产内容（必须复用长内容源的视频/图文/代码三个源头）
- **新增（2026-05-03）**：
  - 国内自媒体视频 / 文章里展示海外网站截图、海外平台访问操作
  - 国内简介挂海外平台链接（GitHub / X / YouTube 等都不挂）
  - 国内视频里演示英文版个人网站操作
  - 把国内身份和海外社交平台账号做明显关联
