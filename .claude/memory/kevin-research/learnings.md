## 2026-05-21 — ASR 模型选型不能只看单语 benchmark
**情境**：调研中文 ASR 时，Belle-whisper-large-v3-zh 在 AISHELL CER 数据极漂亮（-24~65%），表面是最优解。
**经验**：纯单语 fine-tune 会**牺牲跨语言能力**——Belle 把英文识别精度也"压"成了中文同音字，对 Kevin 这种"中文为主+技术术语英文"的场景反而更差。AISHELL/WenetSpeech 都是纯中文数据集，benchmark 漂亮 ≠ 真实场景好。
**下次怎么做**：调研多语种/code-switching 场景时，**必须找 code-switching 专用 benchmark**（如 CS-Dialogue, SEAME, ASCEND）；纯单语数据集只看趋势，不下结论。同时**永远问一句**："这个 fine-tune 牺牲了什么？"
**适用场景**：ASR / TTS / 翻译 / OCR 等多语言模型选型；任何"在 X 任务上 SOTA"的模型都要追问"在 Y 任务上表现如何"。

## 2026-05-24 — 厂商"支持 N 种 X"宣传数字必须打折
**情境**：调研方言 ASR 时，讯飞宣称支持 "202 种方言"，远超学术共识（汉语方言主要 10 大类 + 数十次级方言）。其他厂商也存在类似情况——腾讯 23 种、阿里 18-22 种、电信 40 种，但都未公开**完整方言名单**。
**经验**：(1) 厂商把"地方口音 / 音变 / 子方言"也算入总数虚增。(2) 商用 API "支持"≠"稳定识别准确率高"，实际可商用稳定的约为宣称值的 50-70%。(3) 完整方言名单一般不公开 → 客户实际可用方言要靠商务沟通或实测确认。
**下次怎么做**：调研含"支持 N 种 X"（语言/方言/格式/平台）的厂商宣传时，**必须问三个问题**：(a) 完整名单是什么？(b) 准确率/精度的具体方言/语种 benchmark 数据在哪？(c) 实际产品 case study 用了其中几种？给客户报价前**绝不能在合同 / 文案里直接引用厂商总数**——只承诺有公开 benchmark 的子集。
**适用场景**：任何"支持 N 种 X"的厂商宣传（ASR 方言数、TTS 音色数、翻译语种数、OCR 字符集数、API 集成平台数等），尤其在帮 Kevin 做客户报价 / 产品可行性评估时。

## 2026-05-24 — "human-in-the-loop 数据飞轮"评估必须拆三块：数据形态、对齐成本、合规
**情境**：客户提"客服转人工 → 录音 → 客服整理文本 → 喂训练 → 模型越来越聪明"的飞轮 idea，听上去聪明，但评估时容易只看"客服愿不愿写"忽略整条 pipeline。
**经验**：评估这类 idea 必须问三层：(1) **数据形态**——客服打字纪要是"摘要"非"逐字稿"，ASR fine-tune 要词级对齐文本，颗粒度不对就是无效数据；(2) **对齐成本**——音频 + 强对齐转写需 2-3× 音频时长的人工，业界标准是录音全留 + 独立标注团队消化，**不该压在一线客服身上**；(3) **合规**——客服录音含敏感个人信息，《个人信息保护法》第 55 条要求训练前 PIPIA 评估 + 用户授权 + 脱敏，2026-01 新《网安法》更严，合同里"数据合规责任"必须明确划归甲方。
**下次怎么做**：评估任何"数据飞轮 / 自学习 / 越用越聪明"类产品 idea 时，先画三表：(a) 数据形态→训练价值映射；(b) 飞轮速度计算（含 35-55% 折损系数）；(c) 合同风险清单（含数据所有权 + 模型改善量化节点 + 合规责任划归）。**永远把"每月真实可训练数据量 × 12 / 24 个月"算出来给甲方看**，避免画饼充饥。
**适用场景**：所有 B 端 AI 产品评估、含"客户数据反哺模型"承诺的项目报价、客服/电销/IVR 场景智能化方案、任何含"持续优化 / 越用越准"软承诺的合同审查。

## 2026-05-21 — 推理速度数据要分清"论文"和"实测"
**情境**：SenseVoice 论文宣称比 Whisper-large-v3 快 15×，但 whispernotes 博客 M4 Pro 实测 27 分钟播客对比，速度差距远没那么夸张。
**经验**：模型方论文里的速度对比常用 A100 + 大 batch + 优化推理代码，普通用户在 Mac/单机上跑差距会缩小到 2-5×。下结论"X 比 Y 快 N 倍"前**必须找消费级硬件实测**。
**下次怎么做**：调研模型时，论文速度 ≠ 用户速度，标注清楚"论文数据" vs "M4/3090 实测"；推荐时给"理论上限 + 现实预期"两个数。
**适用场景**：所有 ML 模型性能调研（推理速度、显存、吞吐量）。
