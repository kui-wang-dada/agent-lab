## 0. v1.1 与 v1.0 的核心差异（先讲变化）

v1.0 把需求理解成"通用方言识别 + 文案生成 SaaS 工具"，假设面向 toC 创作者或泛 B 端。
v1.1 收到关键信息——**甲方真实场景是替代人力客服**——产品被重新定义为：

| 维度 | v1.0 | v1.1 |
|---|---|---|
| 产品类别 | 工具型 SaaS（"识别 → 出文案"） | **业务系统**（"识别 → 应答 / 转人工 → 闭环"） |
| 用户 | 创作者 / 银发用户 / 泛 B | **B 端客服中心 + 终端来电方** |
| 核心循环 | 一次性输入输出 | **多轮对话 + 转人工 + 数据回流 + 模型升级**（飞轮） |
| 输出形态 | 文案文本 | **应答语音（TTS）/ 文字回复 + 工单 + 通话记录** |
| 价值主张 | 帮人写得更快 | **降低人力客服成本 + 长期持续变聪明** |
| 报价区间 | ¥3-35w | **¥15-60w 起步 + 飞轮阶段 ¥10-25w/季 + 长期分润可谈** |

**最重要的认知转变**：这不再是"做完即交"的项目，而是一个**长期演进的系统**，Kevin 与甲方的合作关系也从"外包"变成"持续运营/共建"。这同时意味着——更高客单价、更高粘性、但也**更大的承诺负担 + 与 Kevin 时间分配硬约束的紧张**。本文 §4、§5 会反复回到这一点。

## 1. 产品定位选项（让 Kevin 拍板）

三个表述各有侧重，区别在于"谁是用户、价值卖点是什么、合同怎么写"：

### 选项 A：「方言 AI 坐席」(Dialect AI Agent)

- **定位**：替代/分担人工客服坐席，AI 直接接电话/在线对话，处理标准问题
- **卖点**：每个 AI 坐席 = 替代 0.5-2 个人力，按月节省 ¥3-8k/席
- **风险**：承诺过强，效果不达期望会被退货
- **适合**：甲方已有明确 KPI（如"30% 通话由 AI 完成"）+ 容忍 AI 的边界

### 选项 B：「方言客服增强系统」(Dialect Service Augmentation)【**推荐**】

- **定位**：辅助人工客服——实时翻译方言、推荐应答话术、自动生成工单/小结
- **卖点**：让普通话客服也能接方言电话；每席效率提升 20-40%
- **风险**：人在回路，效果差也能"人兜底"，承诺更安全
- **优势**：天然适配 Kevin 提出的"转人工 + 录音回流"飞轮——人工本来就在闭环里
- **适合**：甲方有现成客服团队、想"提质"而非"砍人"

### 选项 C：「方言智能客服中台」(Dialect Customer Service Platform)

- **定位**：完整客服系统（IVR + ACD + 工单 + 知识库 + AI 全套）
- **卖点**：一站式替换现有客服平台
- **风险**：Kevin 一人扛不动客服中台级工程量，必然外包 + 项目失控
- **不推荐**：除非甲方愿意付 ¥100w+ 且接受 6-9 月工期

**Kevin 应在第一次谈话拍板选 B**。理由：
1. 与"飞轮 idea"天然契合——人工录音是产品流程的一部分而非外挂
2. 风险与承诺平衡最好——AI 不灵时不会"客户骂街"，只是"客服多干一点"
3. 报价空间合理（¥15-35w MVP），不至于把 Kevin 拖进客服中台深坑
4. 留出 v2 升级路径：飞轮跑半年后效果好，可以**自然升级为 A**（AI 直接接更多场景）

**飞轮三个发动机**（缺一不可）：
1. **真实通话量**（每月 1k+ 通才有效数据沉淀）
2. **人工标注产能**（谁标？多少时间？质检谁做？——下方 Q3-Q6 必问甲方）
3. **训练 + 上线机制**（LoRA 增量 + 灰度发布；ASR 必须走开源路线，云 API 不开放权重微调，见 research 文档）

### 必答 5 问（不答不报价）

**Q1【最关键】甲方是哪类客服 + 行业？席位规模？**
- 为什么：决定产品形态（电销 vs 客服中心 vs 政务热线）+ 决定合规要求 + 决定报价上限
- 选项：
  - A. **电销外呼**（保险/金融/教培）—— 容忍度高，AI 直接接最容易
  - B. **客服中心 inbound**（电商售后/物流）—— 标准场景，飞轮容易跑
  - C. **政务热线 12345 类**—— 与 Kevin "不接政务" 硬约束冲突，**触发拒签**
  - D. **运营商/银行客服**—— 资质门槛高（等保三级、金融合规），慎接
- 必须问：**席位数（5? 50? 500?）+ 月通话量（1k? 10w? 100w?）+ 当前每席月成本**

**Q2【最关键】方言占比与优先级**
- 为什么：决定 MVP 优先支持哪几种方言、决定能否承诺效果
- 必须答：**TOP 5 方言 + 各自月通话占比**（如"粤语 40% / 闽南 20% / 客家 10% / 普通话夹方言 20% / 其他 10%"）
- 配套问：业务所在省份？目标终端用户分布省份？
- 推荐对策：MVP 只覆盖 TOP 3-5 + 占比 ≥80% 的方言；其余转纯人工

**Q3【飞轮关键】数据所有权 + 标注产能**
- 为什么：飞轮没有"持续标注"就转不起来——这是 Kevin 的产品 idea 能否成立的核心命门
- 必须答的子问题：
  - 通话录音版权归甲方还是来电方？（涉及隐私 + 是否可用于训练）
  - **谁来做事后整理**？现有客服 + 兼职？专职标注？外包？
  - 标注人天预算：每周 / 每月能投多少人时？（行业经验：50 小时高质量方言数据 ≈ 200-400 人时清洗标注，分摊到 3-6 个月才合理）
  - 是否有质检环节？质检者是谁？
- **如果甲方答"没想过这块"或"你来安排"——警示**：飞轮承诺就成了 Kevin 的负担

**Q4【飞轮关键】"逐步变聪明"的可接受周期 + 验收口径**
- 为什么：模型升级见效慢（首批数据需 3-6 个月才足量），甲方期望必须对齐
- 必须答：
  - 上线后多久客户开始期望"看到 AI 进步"？（3 月？6 月？12 月？）
  - 验收用哪个硬指标？候选：CER 下降曲线 / AI 接管率 / 平均通话时长 / 客户满意度
  - "如果 6 个月模型没明显进步"甲方的容忍方案是什么？

**Q5【商务关键】现有客服系统是什么？需不需要集成？**
- 为什么：决定工程边界。客服中台（Genesys / 容联 / 阿里云通信 / 天润融通 / 自研）API 千差万别，集成工作量可能比 AI 本身还重
- 必须答：
  - 现有系统名称 + 是否提供 API？
  - 是 Kevin 做"独立系统 + 接他们" 还是"嵌入他们" 还是"完全替换"？
  - 通话录音怎么拿到？实时流式 SIP 还是事后批量下载？
- **触发拒签**：如果要"完全替换现有客服中台"——选项 C 工程量，不接

### 可后答 6 问（影响细节但不阻塞报价）

**Q6 部署形态：SaaS / 私有云 / 客户内网**（影响交付方式 + 是否能用云 ASR）

## 4. 三阶段产品架构（v1.1 推荐切片）

### Phase 1 — MVP「客服辅助系统」(2-3 个月 / ¥18-35w)

**核心交付物**：
- 实时方言 ASR（覆盖 5-8 个 TOP 方言，用 Qwen3-ASR-Flash）
- 客服面板（Web）：实时方言→普通话字幕 + 推荐话术 + 通话录音回放 + 一键生成通话小结/工单
- 简易 AI 应答（仅 5-10 种高频标准问题，置信度高时自动答，否则转人工）
- 通话归档系统：音频 + 转写 + 元数据入库，**为飞轮做数据采集准备但不做训练**
- 管理后台：通话量统计、识别准确率监控、转人工率统计

**不含**：
- 模型训练 / 飞轮 pipeline（Phase 2）
- 私有化部署
- 完整 IVR / ACD（如客户没有现成系统，建议他们买现成）
- 复杂业务集成（CRM / 工单系统对接，只做开放 API 接口）

**依赖前置**：
- 甲方提供测试用方言录音样本（≥50 条，验证 ASR 准确率）
- 甲方明确客服系统集成方式（Q5 答复）
- 合同里"覆盖方言 + 验收指标"写死

**关键风险**：
- ASR 准确率在客户真实场景可能比厂商口径低 10-15%（环境噪音、口音重）—— 必做"真机测试"再签合同
- AI 应答覆盖率低（业务标准问题没整理清楚） → MVP 设计上**容忍很低应答率**（10-20% 即合格），主打"辅助"卖点

**Kevin 单人 vs 外包占比**：
- Kevin 独立可做：70%（前后端、ASR/LLM 集成、客服面板）
- 需要外包：30%（电话系统 SIP/PSTN 集成——除非客户已有现成系统直接接 API；客服 UI 视觉设计——找 kevin-designer 协作）

### Phase 2 — 飞轮启动「持续学习系统」(+3-6 个月 / ¥15-25w 一次性 + 后续按季)

**核心交付物**：
- 数据 pipeline：通话录音 → 自动清洗（VAD/去静音）→ 推送给人工标注 → 入训练集
- 标注工具：Web 标注台（音频对齐文字 + 方言标签 + 意图标签），供甲方标注团队用
- 增量微调系统：基于 FunASR / SenseVoice / TeleSpeech 开源模型 LoRA 适配（**ASR 训练必须开源路线**，云 API 不开放）
- 月度模型升级流程：影子模式 → A/B 测试 → 灰度发布 → 全量上线
- 效果监控看板：每月 CER 改善曲线、各方言 case 漂移分析、AI 接管率变化

**依赖前置**：
- Phase 1 已上线 ≥3 个月 + 累积通话 ≥10k 条
- 甲方组建标注团队（2-3 人 × 半 fte）或购买外部标注服务
- 计算资源：Kevin 本地 M4 Max 64GB 跑 LoRA 微调够用；如客户要求私有化训练 → 加 ¥3-5w GPU 服务器

**关键风险**：
- **数据量不达标**：通话太少 / 标注跟不上 → 6 个月模型不见进步 → 客户失望。**必须在合同里写"数据飞轮见效以甲方提供 ≥X 小时标注数据为前提"**
- **训练效果有上限**：参考 research 文档结论——50 小时数据 CER 降 10-25%，100+ 才接近商用。**承诺"逐步变聪明"但不承诺"达到 X%"**
- **运维负担**：模型上线后 case 漂移监控需持续投入，Kevin 自己扛不动 → 建议合同里包"按季运维费 ¥3-5w/季"

**Kevin 单人 vs 外包占比**：
- Kevin 独立可做：60%（pipeline 工程、LoRA 训练脚本、监控看板）
- 需要协作：40%（标注工具 UI 找 designer；如客户深度训练需求 → 找外部 ML 工程师，Kevin 抽 PM 费）

### Phase 3 — 规模化「多方言并行 + 私有化」(选做 / ¥30-60w / 6-12 月)

**核心交付物**：
- 覆盖方言扩展到 15-25 种（含温州/闽东等长尾）
- 主动学习：自动识别"模型不会的方言/词"推送给标注队列优先处理
- 客户私有化部署包：Docker compose + 一键部署 + 监控告警
- 多租户 SaaS（如甲方想"二次卖给他们的客户"）

**关键判断：这阶段 Kevin 是否还接？**

**Kevin 建议不要独立接 Phase 3**，理由：
- 工程量超出 Kevin 单人产能（即便有 AI 协助）
- 与 Upwork 主线（60% 时间分配）冲突
- 与"个人 freelance"模式不匹配——更适合开公司

**推荐对策**：Phase 2 结束时与甲方谈判：
- 选项 A：Kevin 退场，引荐外部团队接 Phase 3，Kevin 抽 5-10% 持续顾问费
- 选项 B：Kevin 做"技术合伙人"持股不持工，按需提供架构 review
- 选项 C：如果甲方愿意付 ¥80w+ 并接受 12 月工期，Kevin 拉一个 2-3 人小团队（含 ML 工程师 + 后端 + 全栈）

## 5. 商业模式建议

### 计费结构（推荐"基础 + 飞轮 + 分润"三层）

| 层 | 形式 | 金额 | 周期 |
|---|---|---|---|
| **基础授权费** | 一次性买断 Phase 1 系统 | ¥18-35w | 3 月内付清（30/40/30 三笔） |
| **飞轮订阅费** | Phase 2 启动 + 持续运营 | ¥3-5w/季 | 按季滚动，含每月模型升级 + 监控 + 二线支持 |
| **变动费用** | 按通话分钟（如客户量大）或按席位（如客户量稳定） | ¥0.05-0.15/分钟 OR ¥200-500/席/月 | 按月结算 |
| **训练加价**（可选） | 每新增方言 LoRA 训练 | ¥5-15w | 按方言 |
| **分润**（建议争取） | AI 接管率达到 ≥X% 后，按节省人力成本的 10-20% 抽成 | 浮动 | 按季对账 |

**Kevin 取舍**：
- **必含基础授权费**（保 Kevin 现金流）
- **必含飞轮订阅费**（绑定持续关系 + 避免甲方 6 月后白嫖）

### "智能化逐步提升"硬指标怎么写

**承诺什么 / 不承诺什么**：

- ✅ 承诺：**每月发布一次模型升级版本**（含报告：本月哪些方言新增多少标注数据 / CER 测试集结果对比 / 新增覆盖词汇 / 已知遗留 case）
- ✅ 承诺：**Phase 1 上线 6 月后总体 CER 较 baseline 改善 ≥10%**（前提是甲方提供 ≥50 小时标注数据）
- ❌ 不承诺：**特定 CER 数值**（如"必须达到 15%"——做不到就违约）
- ❌ 不承诺：**特定方言达到商用质量**（温州 / 客家细分等长尾，老实告诉客户"按数据量尽力而为"）

参考 research 文档第 234 行："训练让模型更聪明，但前提是您能提供足够多、足够干净的对齐数据"——把球踢回甲方。
